# ADR-0001: Use Nuxt 4 and Vue 3 as Core Framework

## Status

Accepted

## Context

The project is a “DnD reference hub” web application focused on fast access to reference content during a game session. The MVP has no authentication or server-side user state; core content is delivered from a headless CMS. Key requirements include:

- high rendering and loading performance (TTFB / INP / LCP) for “lookup during session” scenarios;
- SEO and link sharing for reference pages;
- RU/EN support (UI + content);
- strict typing (TypeScript strict) and demonstration of mature engineering practices;
- ability to evolve into V1/V2 (auth, notes, GM board) without rewriting architectural foundations.

Assumptions:

- The application is deployed on a platform compatible with Nuxt 4 (e.g., Vercel with Nuxt preset, or a Node server) and supports edge/CDN caching.
- Most pages are content-driven, where server-side rendering and caching provide tangible value.
- The team/author is working within the Vue 3 ecosystem.

## Decision

- Use Nuxt 4 as the core routing and rendering framework.
- Use Vue 3 Single File Components (SFCs) with `<script setup>` for all UI components; server-side rendering is the default.
- Adopt a “server-first” model: data fetching via `useAsyncData` / `useFetch` on the server wherever it reduces client-side JavaScript and improves first render.
- Use Nitro (Nuxt's server engine) for server-side logic, API routes (`server/api/`), and caching.

## Alternatives Considered

1. **Next.js App Router + React Server Components**
   - Why considered: production-standard stack for SSR with strong caching primitives and a large ecosystem.
   - Why not chosen: Vue 3 / Nuxt 4 better fits the project's framework preference and ecosystem direction.
2. **Nuxt 4 with static-only generation (no SSR)**
   - Why considered: simpler deployment with no server required.
   - Why not chosen: SSR provides better flexibility for future features (auth, personalization, webhook-based revalidation).
3. **Remix**
   - Why considered: strong data-loading patterns and SSR model.
   - Why not chosen: React-based; inconsistent with the Vue 3 direction of this project.
4. **Astro with Vue islands**
   - Why considered: minimal JavaScript and excellent performance for content-heavy projects.
   - Why not chosen: fewer native patterns for future interactive features (auth, notes); additional architectural complexity for V1/V2.

## Consequences

### Positive

- Production-standard stack for SSR, i18n, and content-driven applications in the Vue ecosystem.
- Reduced client-side JavaScript and improved first render for reference pages.
- Nuxt auto-imports for composables and components reduce boilerplate significantly.
- Clear evolution path for future features (auth, server API routes, Pinia state management).
- Nitro provides a unified server layer for caching, API routes, and middleware.

### Negative

- Nuxt 4 SSR requires careful attention to hydration mismatches.
- Caching and revalidation patterns (`routeRules`, `cachedEventHandler`) require familiarity with Nitro.

### Risks

- Risk of hydration mismatches when mixing server-initialized and client-only state incorrectly.
- Risk of accidental caching regressions when modifying `routeRules` or `cachedEventHandler` settings.
- Mitigation: documented architectural rules, review checklists, and baseline performance monitoring.

## Notes (optional)

- The decision assumes “SSR by default” with selective client-only components via `<ClientOnly>` or `.client.vue`.
- State management for V1 will use Pinia (the recommended Vue 3 store, integrated into Nuxt via `@pinia/nuxt`).
- The project uses **Vue Router 5** (shipped with Nuxt 4). Key differences from Vue Router 4: `useRoute()` returns a readonly route object (mutations via `useRouter().push/replace` only); some navigation guard signatures changed. All routing code must be written against the Vue Router 5 API.
- Application source code follows the **Nuxt 4 `app/` directory convention**: `app/pages/`, `app/layouts/`, `app/components/`, `app/composables/`, `app/assets/`. Server code (`server/`) and static files (`public/`) remain at the project root.

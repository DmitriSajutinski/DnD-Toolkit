# ADR-0001: Use Next.js App Router and React Server Components

## Status
Accepted

## Context
The project is a “DnD reference hub” web application focused on fast access to reference content during a game session. The MVP has no authentication or server-side user state; core content is delivered from a headless CMS. Key requirements include:
- high rendering and loading performance (TTFB / INP / LCP) for “lookup during session” scenarios;
- SEO and link sharing for reference pages;
- RU/EN support (UI + content);
- strict typing (TypeScript strict) and demonstration of mature engineering practices;
- ability to evolve into V1/V2 (auth, notes, GM board) without rewriting architectural foundations.

Assumptions:
- The application is deployed on a platform compatible with Next.js (e.g., Vercel or equivalent) and supports edge/CDN caching.
- Most pages are content-driven, where server-side rendering and caching provide tangible value.
- The team/author is prepared to maintain App Router discipline (clear server/client boundaries).

## Decision
- Use Next.js App Router as the core routing and rendering framework.
- Use React Server Components (RSC) by default for content pages; apply client components selectively for interactivity (search UI, local favorites, etc.).
- Adopt a “server-first” model: data fetching and HTML generation on the server wherever it reduces client-side JavaScript and improves first render.

## Alternatives Considered
1) **Next.js Pages Router**
    - Why considered: mature approach with many familiar patterns and examples.
    - Why not chosen: limited architectural evolution compared to App Router; weaker expression of server-first and server/client boundaries; less representative of modern Next.js practices.
2) **Remix**
    - Why considered: strong data-loading patterns and SSR model.
    - Why not chosen: Next.js ecosystem and hosting practices around content sites and ISR/caching are more aligned with project needs.
3) **Astro with islands**
    - Why considered: minimal JavaScript and excellent performance for content-heavy projects.
    - Why not chosen: fewer native patterns for future interactive features (auth, notes); additional architectural complexity for V1/V2.
4) **Nuxt**
    - Why considered: comparable position in the Vue ecosystem.
    - Why not chosen: the project intentionally demonstrates React/Next.js + TypeScript strict practices.

## Consequences
### Positive
- Production-standard stack for SSR/RSC, i18n, and content-driven applications.
- Reduced client-side JavaScript and improved first render for reference pages.
- Clear evolution path for future features (auth, server actions, API integrations).

### Negative
- Increased cognitive load (server/client boundaries, caching, revalidation).
- Requires architectural discipline to avoid performance regressions or unnecessary hydration.

### Risks
- Risk of mixing server and client components incorrectly, increasing bundle size or duplicating requests.
- Risk of accidental caching regressions when modifying fetch/revalidation settings.
- Mitigation: documented architectural rules, review checklists, and baseline performance monitoring.

## Notes (optional)
- The decision assumes “server components by default” with minimal client boundaries.

# ADR-0004: Content Access Layer via Pure Functions and Composables

## Status

Accepted

## Context

- ADR-0003 established the Content Access Layer principle: CMS-specific code must be isolated; pages and API routes must never call Storyblok directly.
- ADR-0003 proposed implementing that layer with a class-based `IContentRepository` interface and a `StoryblokContentRepository` class.
- Nuxt 4 and Vue 3 do not use classes as the primary code organization mechanism. The ecosystem is built around composables, pure functions, and the Composition API. Introducing a class-based pattern in a Nuxt project creates friction with auto-imports, tree-shaking, and framework conventions.
- Assumptions:
  - The isolation guarantee from ADR-0003 (no direct CMS calls outside the content layer) must be preserved.
  - Unit tests for data transformation logic must not require a Nuxt context or network access.
  - The implementation must be replaceable (different CMS, mock data) without touching pages or composables.

## Decision

- Implement the Content Access Layer as **pure functions** on the server side and **composable wrappers** on the client side. No `class` keyword in `server/lib/content/`.
- The boundary remains the same as ADR-0003: `server/lib/content/storyblok/` is the only place that imports from `@storyblok/nuxt` or calls the Storyblok API. Nothing in `app/` imports from `server/lib/content/` directly.

The specific structure:

- **`server/lib/content/types.ts`** — domain entity interfaces (zero CMS-specific fields).
- **`server/lib/content/errors.ts`** — typed error classes with a `_tag` string discriminant (`ContentNotFound`, `ContentUnavailable`).
- **`server/lib/content/storyblok/client.ts`** — `buildStoryblokConfig(runtimeConfig)`: reads token and version from Nuxt runtime config; called per request.
- **`server/lib/content/storyblok/fetchers.ts`** — one pure async function per content operation. Accepts a config object, locale, and identifiers. Returns a domain entity or throws a typed domain error.
- **`server/lib/content/storyblok/mappers/`** — one pure function per content type. Raw Storyblok story in, domain entity out.
- **`app/composables/`** — thin `useFetch` wrappers. Pages import these; they never call `useFetch` or import server modules directly.

## Alternatives Considered

1. **Class-based `IContentRepository` + `StoryblokContentRepository`** _(ADR-0003 decision)_
   - Why considered: familiar OOP pattern, clear interface contract, straightforward to mock.
   - Why not chosen: not idiomatic in Nuxt 4 / Vue 3; classes are not tree-shakeable; instantiation ceremony adds noise with no benefit when a single implementation exists; mocking is equally simple with plain function signatures.

2. **Direct `useFetch` / `useAsyncData` calls in pages with no intermediate layer**
   - Why considered: minimal code, fastest to write for a small MVP.
   - Why not chosen: scatters Storyblok-specific knowledge across page components; makes CMS migration a full-codebase change; impossible to unit-test transformation logic without a running Nuxt context.

3. **Nuxt server composables (e.g. `useConditionRepository()` returning methods)**
   - Why considered: composable API is consistent across server and client.
   - Why not chosen: server composables carry implicit Nuxt context; plain functions with explicit parameters are simpler, more testable, and more portable.

## Consequences

### Positive

- Mapper functions are pure — they can be unit-tested with Vitest in Node environment, no Nuxt, no network, no mocking framework required.
- Fetchers have explicit dependencies (config, locale, slug) — straightforward to test with fixtures.
- No class boilerplate; files are shorter and easier to read.
- Tree-shakeable: only the fetchers actually called in API routes are bundled.
- Swapping the CMS means rewriting `server/lib/content/storyblok/` only. Types, errors, composables, pages — unchanged.

### Negative

- No formal interface type to enforce that all content operations are implemented when adding a new CMS adapter. Discipline must substitute for compiler enforcement.
- Composables in `app/composables/` are thin wrappers; developers must remember to add a composable whenever a new API route is added.

### Risks

- Fetchers may grow large if multiple content types share complex query logic — mitigate by keeping one fetcher file per content type once the module grows.
- Without an interface, a mock/local implementation for tests must duplicate the function signatures manually — mitigate by exporting a type alias of each fetcher's signature from `fetchers.ts`.

## Notes

- Supersedes ADR-0003. The isolation principle from ADR-0003 is fully preserved; only the mechanism changes.
- `_tag` discriminant on error classes is preferred over `instanceof` because it survives module boundary issues and serialization.
- `buildStoryblokConfig` must never be called at module load time — only inside request handlers — to avoid baking secrets into the module scope.

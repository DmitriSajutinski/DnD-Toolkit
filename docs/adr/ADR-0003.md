# ADR-0003: Introduce Content Access Layer (ContentRepository Abstraction)

## Status

Superseded by [ADR-0004](./ADR-0004.md)

## Context

The application retrieves content from Storyblok. Without abstraction, direct CMS calls in routes and components lead to:

- duplicated logic (locales, preview/published flags, mapping);
- harder testing;
- difficulty changing CMS or caching strategy;
- inconsistent handling of slugs, locales, and fallbacks.

Assumptions:

- MVP requires common read operations: page by slug, lists/indexes, reference lookups, minimal search.
- Preview vs published mode must be handled consistently.

## Decision

- Introduce a Content Access Layer (`ContentRepository` or equivalent) as the single interface for content reads.
- Encapsulate within it:
  - CMS client/API usage;
  - localization and fallback rules;
  - preview vs published logic;
  - mapping from CMS data to domain DTOs.
- Disallow direct CMS access from UI and routing layers.

## Alternatives Considered

1. **Direct CMS calls in server components**
   - Why considered: fastest initial implementation.
   - Why not chosen: duplication, poor testability, difficult caching and evolution.
2. **Using CMS SDK without abstraction**
   - Why considered: less custom code.
   - Why not chosen: SDK does not reflect domain use cases or enforce consistency.
3. **GraphQL/unified data layer**
   - Why considered: strict schemas and unified querying.
   - Why not chosen: excessive complexity and infrastructure for MVP.
4. **Class-based `IContentRepository` interface + `StoryblokContentRepository` class** _(chosen in this ADR, superseded by ADR-0004)_
   - Why considered: familiar OOP pattern, clear contract, easy to mock in tests.
   - Why superseded: class-based patterns are not idiomatic in Nuxt 4 / Vue 3. Pure functions + composables are simpler, tree-shakeable, and equally testable. ADR-0004 documents the replacement.

## Consequences

### Positive

- Consistent content access and localization rules.
- Easier testing via repository mocks.
- Flexibility for future CMS changes or additional caching layers.

### Negative

- Additional abstraction layer requires maintenance.
- Risk of an anemic or poorly defined interface.

### Risks

- Over-abstraction without clear value.
- Leakage of CMS-specific concepts into domain models.
- Mitigation: keep repository methods use-case driven, document contracts, and limit API surface.

## Notes (optional)

- Repository is the natural place for caching and observability hooks.
